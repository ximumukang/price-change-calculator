## 🔴 P0 - 严重安全问题（立即修复）

### 1. [ ] 移除配置文件中的硬编码敏感信息
**风险等级**: 🔴 严重  
**问题描述**: 
- `application.yml` 中硬编码了数据库密码 (`root/root`)
- JWT 密钥使用弱密钥且硬编码在配置文件中
- RSA 私钥直接暴露在配置文件中

**影响范围**: 
- 一旦代码泄露，攻击者可直接访问数据库
- JWT 可被伪造
- RSA 私钥泄露导致加密失效

**修复方案**:
```yaml
# application.yml 改为引用环境变量
spring:
  datasource:
    password: ${DB_PASSWORD}

jwt:
  secret: ${JWT_SECRET}

app:
  rsa:
    private-key: ${RSA_PRIVATE_KEY}
    public-key: ${RSA_PUBLIC_KEY}
```

**操作步骤**:
1. 创建 `.env.example` 文件作为模板（不包含真实值）
2. 修改 `application.yml` 引用环境变量
3. 将 `.env` 加入 `.gitignore`（已存在规则 `*.env`）
4. 生成强随机密钥并设置到环境变量

**验证方法**: 检查 git 历史中是否还有明文密码

---

### 2. [ ] 加强 JWT 密钥强度
**风险等级**: 🔴 严重  
**问题描述**: 
- 当前 JWT 密钥为可读字符串，不符合 HMAC-SHA256 最佳实践
- 应使用至少 256 位（32 字节）的随机密钥

**修复方案**:
```bash
# 生成安全的 JWT 密钥
openssl rand -base64 32
```

**操作步骤**:
1. 使用上述命令生成新密钥
2. 设置为环境变量 `JWT_SECRET`
3. 更新文档说明密钥管理要求

---

### 3. [ ] 启用数据库连接 SSL/TLS
**风险等级**: 🔴 严重  
**问题描述**: 
- `application.yml` 中 `useSSL=false`，数据库通信未加密
- 生产环境中数据可能被窃听

**修复方案**:
```yaml
url: jdbc:mysql://localhost:3306/price_change?useSSL=true&requireSSL=true&verifyServerCertificate=true
```

**操作步骤**:
1. 配置 MySQL SSL 证书
2. 修改 JDBC URL 启用 SSL
3. 测试连接是否正常

## 🟠 P1 - 高优先级安全问题（本周内修复）

### 4. [ ] 添加请求速率限制（Rate Limiting）
**风险等级**: 🟠 高  
**问题描述**: 
- 登录接口无频率限制，易受暴力破解攻击
- API 无限流保护，可能被 DDoS

**修复方案**: 使用 Bucket4j 或 Spring Cloud Gateway RateLimiter

**操作步骤**:
1. 添加依赖 `bucket4j-spring-boot-starter`
2. 配置登录接口限流（如：5 次/分钟）
3. 配置全局 API 限流

**验证方法**: 快速多次请求登录接口，应返回 429 Too Many Requests

---

### 5. [ ] 增强密码策略
**风险等级**: 🟠 高  
**问题描述**: 
- 注册时无密码复杂度要求
- 无密码长度限制

**修复方案**:
```java
// RegisterRequest.java
@NotBlank(message = "密码不能为空")
@Size(min = 8, max = 128, message = "密码长度必须在8-128字符之间")
@Pattern(regexp = "^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d).*$", 
         message = "密码必须包含大小写字母和数字")
private String password;
```

**操作步骤**:
1. 修改 `RegisterRequest.java` 添加验证注解
2. 前端同步更新密码强度提示
3. 编写单元测试验证规则

---

### 6. [ ] 实现账户锁定机制
**风险等级**: 🟠 高  
**问题描述**: 
- 登录失败无次数限制
- 无临时锁定机制

**修复方案**: 
- 连续失败 5 次后锁定账户 15 分钟
- 使用 Redis 或数据库记录失败次数

**操作步骤**:
1. 在 `UserService` 中添加失败计数逻辑
2. 登录成功后清除失败计数
3. 锁定期间拒绝登录尝试

### 7. [ ] 添加 SQL 注入防护审计
**风险等级**: 🟠 高  
**问题描述**: 
- 虽然使用了 MyBatis-Plus，但需确认所有查询都使用参数化
- 检查是否有动态 SQL 拼接

**操作步骤**:
1. 审查所有 Mapper 和 Service 层代码
2. 确保没有使用 `${}` 拼接用户输入

### 8. [ ] 完善 CORS 配置
**风险等级**: 🟠 高  
**问题描述**: 
- `allowedHeaders` 设置为 `*`，过于宽松
- 生产环境应限制具体的请求头

**修复方案**:
```java
configuration.setAllowedHeaders(List.of(
    "Authorization", 
    "Content-Type", 
    "Accept"
));
```

**操作步骤**:
1. 明确列出允许的请求头
2. 根据实际需求调整允许的方法
3. 考虑添加 `maxAge` 缓存预检请求

## 🟡 P2 - 中优先级安全问题（本月内修复）

### 9. [ ] 添加安全响应头
**风险等级**: 🟡 中  
**问题描述**: 
- 缺少常见 HTTP 安全头
- 可能遭受 XSS、点击劫持等攻击

**修复方案**: 添加 SecurityFilterChain 配置
```java
http.headers(headers -> headers
    .contentTypeOptions(Customizer.withDefaults()) // X-Content-Type-Options
    .xssProtection(Customizer.withDefaults())      // X-XSS-Protection
    .frameOptions(FrameOptionsConfig::deny)        // X-Frame-Options
    .cacheControl(CacheControlConfig::disable)     // Cache-Control
);
```

**操作步骤**:
1. 在 `SecurityConfig` 中添加安全头配置
2. 测试浏览器是否正确接收这些头
3. 验证功能不受影响

### 10. [ ] 实现 Token 刷新机制
**风险等级**: 🟡 中  
**问题描述**: 
- Token 过期时间为 2 小时，用户体验不佳
- 无刷新 Token 机制

**修复方案**: 
- Access Token: 15 分钟

**操作步骤**:
1. 设计双 Token 机制
2. 实现 `/api/auth/refresh` 接口
3. 前端自动刷新 Token

### 12. [ ] 增强异常处理安全性
**风险等级**: 🟡 中  
**问题描述**: 
- `GlobalExceptionHandler` 可能泄露堆栈信息
- 生产环境应隐藏技术细节

**修复方案**:
```java
@ExceptionHandler(Exception.class)
@ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
public Result<Void> handleGenericException(Exception ex) {
    log.error("系统内部错误", ex); // 完整日志仅服务端
    return Result.error(500, "服务器内部错误"); // 客户端仅见通用消息
}
```

**操作步骤**:
1. 区分开发和生产环境的错误信息
2. 确保不暴露堆栈跟踪给客户端
3. 添加完整的服务器端日志

---

**修复方案**: 使用 OWASP Java Encoder

**操作步骤**:

2. 对输出到前端的用户数据进行编码

### 20. [ ] 实施最小权限原则
**风险等级**: 🟢 低  
**问题描述**: 
- 数据库用户使用 root 权限过高
- 应创建专用应用用户并限制权限

**修复方案**:
```sql
CREATE USER 'price_app'@'localhost' IDENTIFIED BY 'strong_password';
GRANT SELECT, INSERT, UPDATE, DELETE ON price_change.* TO 'price_app'@'localhost';
```
