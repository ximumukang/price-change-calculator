## 🔴 P0 - 严重安全问题（立即修复）

### 1. [x] 移除配置文件中的硬编码敏感信息
**风险等级**: 🔴 严重  
**问题描述**: 
- `application.yml` 中硬编码了数据库密码 (`root/root`)
- JWT 密钥使用弱密钥且硬编码在配置文件中
- RSA 私钥直接暴露在配置文件中

**影响范围**: 
- 一旦代码泄露，攻击者可直接访问数据库
- JWT 可被伪造
- RSA 私钥泄露导致加密失效

**修复方案**:
```yaml
# application.yml 改为引用环境变量
spring:
  datasource:
    password: ${DB_PASSWORD}

jwt:
  secret: ${JWT_SECRET}

app:
  rsa:
    private-key: ${RSA_PRIVATE_KEY}
    public-key: ${RSA_PUBLIC_KEY}
```

**操作步骤**:
1. ✅ 创建 `.env.example` 文件作为模板（不包含真实值）
2. ✅ 修改 `application.yml` 引用环境变量
3. ✅ 将 `.env` 加入 `.gitignore`
4. ✅ 生成强随机密钥并设置到环境变量

**验证方法**: 检查 git 历史中是否还有明文密码

**操作步骤**:
1. ✅ 使用上述命令生成新密钥
2. ✅ 设置为环境变量 `JWT_SECRET`
3. ✅ 更新文档说明密钥管理要求

### 16. [x] 实现 API 签名验证

### 19. [x] 添加 CSRF 防护（如需 Cookie 认证）
**风险等级**: 🟢 低  
**问题描述**: 
- 当前禁用 CSRF（因为使用 JWT）
- 如改用 Cookie 需重新启用

**评估结果**: 
当前项目使用 JWT + Authorization Header 认证方式，CSRF 攻击无法跨域伪造自定义请求头，因此 CSRF 风险较低。

**当前配置**:
```java
// SecurityConfig.java
.csrf(csrf -> csrf.disable())  // 已明确禁用 CSRF（JWT 认证无需此保护）
```

**未来建议**:
如果将来改为 Cookie-based 认证（如 HttpOnly Cookie 存储 Token），需要：
1. 重新启用 CSRF 保护：`.csrf(csrf -> csrf.enable())`
2. 使用 `CookieCsrfTokenRepository` 管理 CSRF Token
3. 前端在表单或请求头中携带 CSRF Token

**结论**: 当前架构下无需额外 CSRF 防护措施。
